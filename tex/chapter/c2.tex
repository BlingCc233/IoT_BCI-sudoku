\chapter{相关研究与理论基础}

\section{BCI-IoT 通信需求与评价维度}
BCI-IoT 链路同时承载连续脑电数据与控制指令，具有“高敏感数据、低时延交互、长期在线运行”的复合特征\cite{lei2017bcireview,hua2018iotsecurity,sicari2015iot}。Sicari 等在 IoT 安全综述摘要中强调，``Traditional security countermeasures cannot be directly applied to IoT technologies due to the different standards and communication stacks involved''\cite{sicari2015iot}。因此，协议评价不能只看“是否加密”，还需同时回答三个问题：其一，安全机制是否完整；其二，资源开销是否可控；其三，外观特征是否会被侧写模型稳定学习。

本文据此采用四个评价维度：安全建立过程、会话保护能力、资源占用与外观可识别性。后续对比均在统一消息规模、统一统计定义与重复实验中位数规则下开展。该评估准则与 IoT 场景中关于 TLS/DTLS 性能研究所强调的“同负载、同实现、同统计”原则一致\cite{restuccia2020lowpower,gallenmueller2019dtls}。

\section{主流协议通信机理与安全实现}
\subsection{Pure-AEAD 基线}
本文所称 Pure-AEAD（纯记录层 AEAD）是“仅记录层加密”的最简方案。认证加密（Authenticated Encryption with Associated Data, AEAD）是一种同时提供数据机密性、完整性和真实性的加密模式\cite{rfc5116}。形式化地，一个 AEAD 方案由三个算法组成：密钥生成算法 $\mathcal{K}$、加密算法 $\mathcal{E}$ 和解密算法 $\mathcal{D}$。
RFC 5116 在引言中直接指出，认证加密 ``in addition to providing confidentiality ... provides a way to check its integrity and authenticity''（Section 1）\cite{rfc5116}。该原始定义明确了 AEAD 的两类核心安全服务：机密性与完整性/真实性校验。

加密过程可表示为 $C = \mathcal{E}_K(N, A, P)$，其中 $K$ 为密钥，$N$ 为随机数（Nonce），$A$ 为关联数据（Associated Data），$P$ 为明文。解密过程表示为 $P/\bot = \mathcal{D}_K(N, A, C)$，若验证失败则输出符号 $\bot$。在 Pure-AEAD 方案中，通信双方预共享密钥（PSK）派生会话密钥与盐值，再按记录格式逐条执行 AEAD 加密/解密。在该路径中，机密性与完整性来自 AEAD 标签验证；但身份认证、新鲜性与重放窗口等能力不由记录层接口自动提供，而需上层协议自行补足。其工程优点是实现轻、状态少、吞吐高；局限是安全职责集中在上层状态机，设计不当时更易出现重放或身份混淆边界。

\subsection{MQTT/TLS 基线}
MQTT（Message Queuing Telemetry Transport）是一种基于发布/订阅（Publish/Subscribe）范式的轻量级消息传输协议，专为低带宽、高延迟或不可靠网络环境设计\cite{mqtt311}。MQTT 协议的核心机制包括主题（Topic）匹配与服务质量（Quality of Service, QoS）分级。QoS 级别分为三类：QoS 0（At most once，至多一次）、QoS 1（At least once，至少一次）和 QoS 2（Exactly once，恰好一次），分别对应不同的可靠性保障与资源开销。
MQTT v3.1.1 标准摘要对其定位的原文表述为：``MQTT is a Client Server publish/subscribe messaging transport protocol. It is light weight, open, simple''（Abstract）\cite{mqtt311}。

在安全实现上，MQTT 通常依赖下层传输层安全协议（Transport Layer Security, TLS）提供加密通道。本文基线实现过程为：设备端与服务端先建立 TCP 连接，再完成 TLS 1.3 握手，随后执行 MQTT 的 \texttt{CONNECT/CONNACK}、\texttt{SUBSCRIBE} 与 \texttt{PUBLISH} 交互\cite{rfc8446,zheng2023mqtttamarin}。郑红兵等指出，当前 MQTT 协议安全性分析 ``主要面向协议实现平台，缺少面向协议标准的安全性测试''，并报告其形式化验证中出现 ``9 种保密属性违反和 29 种认证属性违反''\cite{zheng2023mqtttamarin}。TLS 负责证书认证、密钥协商与记录加密，MQTT 负责主题路由与消息分发语义。该路线工程成熟、生态丰富，但系统由“传输安全层 + 消息中间件层”叠加构成，实现链路长，且加密后外观特征仍可被侧写模型利用。

\subsection{CoAP/DTLS 基线}
CoAP（Constrained Application Protocol）是为受限节点和网络设计的专用 Web 传输协议，旨在实现类似 HTTP 的 RESTful 交互模型\cite{rfc7252}。CoAP 运行在 UDP 之上，定义了四种消息类型：可确认消息（Confirmable, CON）、不可确认消息（Non-confirmable, NON）、确认消息（Acknowledgement, ACK）和复位消息（Reset, RST）。通过消息 ID 和 Token 机制，CoAP 实现了轻量级的请求/响应匹配与重传控制。
RFC 7252 对 CoAP 的直接定义是 ``a specialized web transfer protocol for use with constrained nodes and constrained networks''（Section 1）\cite{rfc7252}，这也是本文将其作为受限网络基线的重要依据。

为获得加密与认证，CoAP 常与数据报传输层安全协议（Datagram Transport Layer Security, DTLS）叠加使用。DTLS 在 TLS 的基础上增加了抗丢包与重排序机制（如 Cookie 交换、序列号窗口），以适应不可靠的数据报传输\cite{rfc9147}。典型流程为：先完成 DTLS 握手，再承载 CoAP 业务报文。任智等在泛在电力物联网场景指出，``接入节点数多、设备资源受限''会带来拥塞与时延问题，且其仿真结果表明改进算法可提升吞吐和成功率\cite{ren2020coapcongestion}。该方案对短报文友好，但公网策略、NAT 与防火墙常使 UDP 可达性成为首要约束；在受限网络中，协议可达性问题往往先于密码学机制问题暴露\cite{rfc8085,rfc4787,li2011nat,ren2020coapcongestion}。

\subsection{Sudoku-pure 方案}
Sudoku-pure 采用“认证握手 + 会话记录 + 外观层编码”三层结构。握手阶段使用证书签名、临时密钥协商、摘要绑定与确认码，保障身份、密钥一致性和握手新鲜性；会话阶段使用 AEAD 记录层；最外层将字节编码为数独提示字节序列，并可注入填充字节以改变外观统计特征。该方案定位于 BCI-IoT 场景下“主动攻击面压缩 + 侧写可学习性抑制”的联合优化。

\subsection{协议状态机与安全职责分层}
从实现责任看，四类路线在“安全能力由谁提供”上存在明显差异：
\begin{itemize}
  \item Pure-AEAD：记录层负责机密性/完整性；身份认证与防重放由上层自管；
  \item MQTT/TLS：TLS 负责握手安全与密钥；MQTT 负责会话语义与路由；
  \item CoAP/DTLS：DTLS 负责握手与会话保护；CoAP 负责报文匹配与重传；
  \item Sudoku-pure：握手状态机提供身份与新鲜性，记录层提供机密性，外观层提供抗侧写调控。
\end{itemize}

这一区分对于论文判据有直接影响：Roman 等在分布式 IoT 综述摘要中指出，需同时评估体系优势与 ``security and privacy challenges''\cite{roman2013distributediot}；刘奇旭等也指出，``物联网访问控制一旦被打破，将造成隐私数据泄露、权限滥用等严重后果''\cite{liu2022accesscontrol}。因此，当“重放可拒绝、外观可调控、资源有上界”是联合目标时，单层机制通常不足，需在状态机、记录层与外观层之间显式分工\cite{roman2013distributediot,sicari2015iot,liu2022accesscontrol}。

\section{IoT 安全协议性能实证与研究空缺}
TLS/DTLS 在受限设备上的性能代价已有较多实证。Restuccia 等在低功耗 IoT 设置中比较 TLS/DTLS 版本与配置，其摘要指出 ``we measure and compare bytes-over-the-air, memory footprint, and energy consumption''\cite{restuccia2020lowpower}；Gallenm{\"u}ller 等则强调 DTLS 在工程侧的成本可建模，并在摘要中指出 ``we construct a model to determine the performance of generic DTLS-enabled applications''\cite{gallenmueller2019dtls}。这些研究为本文第 4 章的性能判据提供了经验依据。

但现有工作仍存在三点空缺。其一，多数研究聚焦“加密成本”，较少将“流量外观可学习性”与性能指标并行讨论；其二，公开评测通常停留在实验室网络，缺少受限公网条件下的复核；其三，关于 BCI 场景的研究更多关注算法模型与数据安全\cite{meng2024adversarial,lei2017bcireview}，较少提供“协议机制--实现--攻击验证”闭环证据。本文将这三类空缺作为后续章节的重点补全目标。

\section{流量侧写与对抗研究}
流量分析（Traffic Analysis）是指攻击者在无法解密加密流量内容的情况下，通过观察数据包的外部特征（如长度、时间间隔、方向序列等）推断通信内容、用户身份或应用类型的攻击技术\cite{velan2015survey}。根据攻击目标的不同，流量分析可分为协议识别（Protocol Identification）、网站指纹识别（Website Fingerprinting）和用户行为分析（User Behavior Analysis）。
Velan 等在综述中直接指出，加密连接的初始化与协议结构仍会 ``give away a lot of information for encrypted traffic classification and analysis''\cite{velan2015survey}。这一定性结论与本文后续侧写实验观察一致。

在加密流量中，虽然载荷内容被隐藏，但协议交互产生的侧信道特征往往难以完全消除：
\begin{itemize}
  \item \textbf{包长特征（Packet Size）}：不同应用层协议或业务逻辑产生的数据包大小分布具有显著差异，通过分析包长序列（Packet Size Sequence）可有效区分视频流、网页浏览与即时通讯流量\cite{pan2016encryptedtrafficcn}；
  \item \textbf{时序特征（Inter-arrival Time, IAT）}：数据包到达的时间间隔反映了应用层的交互频率与处理延迟，是识别交互式应用的重要特征；
  \item \textbf{方向特征（Direction）}：上下行流量的比例与突发模式（Burstiness）构成了流量的方向性指纹。
\end{itemize}
TLS 1.3 规范同样给出关键边界：``TLS does not hide the length of the data it transmits''（RFC 8446, Section 1）\cite{rfc8446}。因此，长度与时序特征仍可构成可学习信号。
Pan 等在综述摘要中也指出，现有方法受 ``traffic camouflage technology, new protocols of HTTP/2.0 and QUIC'' 等因素影响\cite{pan2016encryptedtrafficcn}。

现有研究已证明，即使明文不可见，上述统计特征仍可支持高精度分类\cite{hou2024encmalware}。侯剑等将被动检测定义为 ``在用户无感知且不执行任何加密或解密操作的前提下，识别加密恶意流量''\cite{hou2024encmalware}。针对 BCI-IoT 场景，攻击者可利用这些特征推断用户当前的脑电状态（如专注、放松）或控制指令类别。本文采用与 Aparecium 类工具一致的流程：先抽取长度与序列特征，再执行协议识别与类别恢复评估。

此外，本文引入 TLS 中间人代理解密对照，用于刻画特定运维/攻防条件下终止代理点的可见性风险，即“链路加密已启用但业务仍可恢复”的现实边界\cite{wilkens2021passivetls}。该场景与 Wilkens 等的实验观察一致：其原文报告 ``TLS decryption increases a runtime overhead of about 2.5 times''，表明在具备运维能力的观测点，业务可见性与计算代价可以被同时工程化\cite{wilkens2021passivetls}。

\section{Sudoku-pure 数学基础与术语}
\subsection{术语定义}
为减少歧义，本文统一采用以下中文术语：
\begin{itemize}
  \item \textbf{提示字节}：编码后可被解码器识别为“数独提示位”的字节；
  \item \textbf{填充字节}：仅用于外观扰动、解码时被忽略的字节；
  \item \textbf{模式模板}：定义字节位上“冗余位/位置位/取值位”分布的 8 位模式；
  \item \textbf{解码键}：4 个提示字节按序拼接得到的 32 位键值。
\end{itemize}

在实现与论文叙述中，本文将“提示字节/填充字节/模式模板”界定为功能术语。安全性论证建立在标准握手、密钥派生、AEAD 与状态机正确性之上，外观编码用于外观调控而非密码学能力替代。

4$\times$4 Sudoku 解空间定义为
\[
\mathcal{S}=\{g\mid g \text{满足行、列与 }2\times2\text{ 子宫格约束}\},
\]
其规模为 $|\mathcal{S}|=288$，可选取其中 256 个解与字节集合建立一一对应\cite{oeisA107739,oeisA109252}。

\subsection{组合空间与排列基数}
从组合数学角度看，若按会话种子从 288 个可行网格中为 256 个字节选择互异映射，则可行映射上界为排列数
\[
P(288,256)=\frac{288!}{32!}\approx 2^{1825}.
\]
该值用于刻画外观映射重排的数量级上界，可反映离散映射空间规模。进一步地，每个字节的提示位位置组合满足
\[
|\mathcal{C}|=\binom{16}{4}=1820,
\]
与填充注入共同作用时，会显著扩大可观测外观形态数量。该分析与 Sudoku 相关组合研究“搜索空间极大、约束主导求解复杂度”的结论一致\cite{lynce2006satsudoku}.

\subsection{可逆映射与复杂度}
对字节 $b\in\mathcal{B}$，先映射到网格 $g_b=f_k(b)$，再从网格中抽取 4 个提示位编码为 4 个提示字节。解码时按 4 字节拼接形成解码键，通过映射表 $D$ 还原原字节。构表阶段若发现不同原字节产生同一解码键，则直接拒绝构表，因此可保证单值可逆。

在 ASCII 布局下，提示字节可写为
\[
h = 0x40 \,|\, (v \ll 4)\,|\, p,
\]
其中 $p\in[0,15]$ 为位置编码，$v\in[0,3]$ 为取值编码。编码与解码对长度为 $n$ 的消息均为线性复杂度 $O(n)$。

无填充时，若第 $i$ 个字节候选提示组数量为 $K_i$，可观测形态规模为
\[
N_{\text{无填充}}=\prod_{i=1}^{n}K_i\approx 8^n.
\]
若每字节存在 5 个填充注入检查点，报文尾部 1 个检查点，填充池大小为 $M$，则
\[
N_{\text{线上}} = N_{\text{无填充}}(M+1)^{5n+1}.
\]
该表达式解释了“外观随机性增强”与“带宽开销上升”之间的必然权衡。

\subsection{Sudoku 握手安全性的比较说明}
与 Pure-AEAD 相比，Sudoku 在握手阶段显式引入证书签名、双向随机数回显、摘要绑定与确认码，安全收益体现在三个方面：
\begin{itemize}
  \item \textbf{身份约束}：握手消息均与证书及签名绑定，降低身份伪造风险；
  \item \textbf{上下文绑定}：ClientHello/ServerHello 摘要进入确认码计算，降低反射与转发攻击成功率；
  \item \textbf{新鲜性约束}：随机数与时间窗口进入重放检测令牌，历史握手难以直接复用。
\end{itemize}

与 TLS/DTLS 相比，Sudoku 的改进重点在于将外观调控能力纳入同一协议状态机，并保持资源上界可控。基于该定位，本文将其归类为“工程折中方案”。

\section{统一评测指标与复现规则}
本文采用如下统一指标：
\[
R_{\text{开销}}=\frac{B_{\text{wire}}}{B_{\text{payload}}},
\quad
\overline{\mathrm{RTT}}=\frac{1}{N}\sum_{i=1}^{N}\mathrm{RTT}_i,
\quad
\mathrm{P95}=\operatorname{Quantile}_{0.95}(\{\mathrm{RTT}_i\}),
\]
\[
H=-\sum_{x=0}^{255}p_x\log_2 p_x,
\quad
\rho_{\text{可打印}}=\frac{\sum_{x=0x20}^{0x7E}c_x}{\sum_{x=0}^{255}c_x},
\quad
T_{\text{payload}}=\frac{B_{\text{payload}}}{t}.
\]
其中 $H$ 为字节熵，$\rho_{\text{可打印}}$ 为可打印字节比例。所有关键结论均要求“多次运行 + 中位数 + 稳定性检查”，避免单次抖动主导结果。
