\chapter{结论与展望}

\section{研究问题回答}
围绕第 1 章提出的 RQ1--RQ4，本文结论如下。

\subsection{RQ1：受限设备可用性是否成立}
在跨国受限网络中，Sudoku-pure 与 Sudoku-packed 的 RTT 分别为 108.676ms 与 105.536ms，与 DTLS（109.040ms）处于同量级，且均低于 MQTT/TLS（150.072ms）；在数据面内存指标上，Sudoku 两条路径分别为 2712B 与 2936B，显著低于 DTLS（185472B）与 MQTT/TLS（97680B）。因此，RQ1 成立：在真实受限部署下，Sudoku 可在资源可控前提下保持可用\cite{restuccia2020lowpower,gallenmueller2019dtls}。

\subsection{RQ2：握手安全机制是否有效}
主动攻击实验显示，重放、篡改与资源探测均进入拒绝路径；TLS MITM 对照中类别恢复率为 1.0，表明链路级加密与业务语义保密之间仍存在可利用风险。Sudoku 将签名认证、摘要绑定、确认码与重放缓存集中在同一状态机，能在工程上提供可验证的握手安全闭环\cite{rfc8446,rfc9147,wilkens2021passivetls}。

\subsection{RQ3：外观编码是否具备可调且可逆特性}
协议识别结果 96.67\% 说明常规加密流量仍可被特征学习；BCI 类别侧写中，Sudoku 无填充恢复率为 1.0000，而填充模式下降为 0.1667。结合第 3 章可逆构表与冲突拒绝机制，表明 Sudoku 外观层能够在“可逆解码”前提下实现可调混淆\cite{velan2015survey,hayes2016kfingerprinting,sirinam2018deepfingerprinting}。

\subsection{RQ4：结论是否可复核}
本文各图表均可追溯到仓库证据目录与字段映射；本地与跨国双环境得到一致趋势：Sudoku 对 DTLS/MQTT 的内存守卫稳定通过，RTT 保持同量级。因此，RQ4 成立，结论具备复核性与可重复性。

\section{总体结论}
本文提出并实现的 IoT\_BCI-Sudoku 协议，不追求单一指标极值，而是追求“安全、效率、抗侧写”三目标可解释平衡。综合证据可得：
\begin{itemize}
  \item \textbf{性能层面}：Sudoku 在跨国受限网络下 RTT 与主流方案同量级，内存占用明显更低；
  \item \textbf{安全层面}：握手抗重放与抗篡改路径有效，异常输入可被拒绝；
  \item \textbf{外观层面}：填充与模板机制可降低类别侧写恢复率，并在 ASCII 模式下改变 DPI 标签；
  \item \textbf{工程层面}：pure 与 packed 形成可配置取舍，分别面向混淆优先与带宽优先场景。
\end{itemize}

据此，Sudoku 可定位为 BCI-IoT 场景下的“工程可用协议”：在受限部署条件下可提供更可控的资源行为与更强的外观调控能力。

\section{与基线方案的客观对比}
为保持学术严谨，本文明确列出 Sudoku 的不足与边界。
\begin{itemize}
  \item 相比 Pure-AEAD：Sudoku-pure 带宽开销更高，窄带场景成本更大；
  \item 相比 MQTT/TLS：缺乏成熟 broker 生态与现成平台适配层；
  \item 相比 CoAP/DTLS：对“必须原生 UDP”的业务语义适配较弱；
  \item 相比标准 TLS/DTLS：标准化程度、第三方互操作证据仍不足。
\end{itemize}

同时，基线协议也存在对应边界：CoAP/DTLS 在公网受限场景下可达性受制于 UDP 路径\cite{rfc7252,rfc9147,rfc7925}；MQTT/TLS 与传统加密隧道仍面临流量侧写识别风险\cite{mqtt311,pan2016encryptedtrafficcn,hou2024encmalware}；Pure-AEAD 路线若无额外握手状态机，重放新鲜性保障较弱\cite{rfc5116}。

\section{应用场景与工程建议}
基于实证结果，Sudoku 更适用于以下场景：
\begin{itemize}
  \item \textbf{远程康复与神经反馈链路}：业务类别隐私敏感，且终端资源受限；
  \item \textbf{边缘控制上行通道}：希望在不改内核 TCP 的前提下增强握手与外观防护；
  \item \textbf{受审查网络下的专用业务流}：通过 ASCII 与模板机制降低被快速规则化标记的概率。
\end{itemize}

在工程部署上，建议按照链路约束选择模式：若优先抗侧写则选 pure；若优先带宽效率则选 packed，并结合业务风险评估设置填充率与模板策略。

\section{未来工作}
后续研究可沿以下方向推进：
\begin{itemize}
  \item \textbf{多地域长期实验}：扩展到更多公网节点，补全跨运营商、跨时段统计；
  \item \textbf{更强攻击模型}：引入深度时序模型与主动探测联动，评估最坏情形；
  \item \textbf{自适应参数控制}：基于链路状态在线调节填充率与 pure/packed 切换；
  \item \textbf{标准化与互操作}：固化报文格式、错误码和一致性测试集，提升跨平台可落地性；
  \item \textbf{与行业协议融合}：探索与 MQTT 语义层或 CoAP 资源模型的协同封装路径。
\end{itemize}

综上，本文完成了从理论、实现到证据的完整闭环，并给出了 Sudoku 方案在真实约束下的优势、成本与边界，为后续扩展和工程落地提供了可复核基础。
